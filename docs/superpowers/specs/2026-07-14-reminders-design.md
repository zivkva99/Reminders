## Problem Statement

You have two working, deeply-iterated personal Android apps — **Shape** (daily exercise
counter with a goal of 5, DataStore-backed, exact-alarm reminders, weekly summary,
12-month heatmap) and **ReadBook** (a 15-minute reading timer habit plus a fully
separate Tanakh chapter-a-day tracker, both Room-backed, with foreground-service
timing, inexact-alarm nudges, notification actions, and heavy TDD discipline). Each
app reinvented the same infrastructure — scheduling, notification channels, streaks,
boot-recovery — from scratch, and adding a new habit to either app means building that
infrastructure a third time. **Reminders** replaces both with one app built around a
small, reusable engine: three habit *kinds* (Counter-with-goal, Timer-with-duration,
Schedule-cursor) that any future habit instance can be built from, plus one unified
dashboard.

## What Makes This Cool

Today, "add a new habit" means writing a new Kotlin app's worth of scheduling,
notification, and stats plumbing. In Reminders, adding a new instance of an existing
kind — a second timer habit, a second counter habit — costs one data row, not one new
class. The generic engine is the actual point: this isn't a bigger habit tracker, it's
an engine that happens to ship with three habits already loaded. On top of that, a
`WorkManager`-based cross-habit evaluator (identified during this session's
independent second opinion, and deliberately pulled into v1 scope rather than
deferred) lets habits react to each other — e.g. escalating one nudge based on
another habit's state — a capability none of the three source apps had, because none
of them knew about each other.

## Constraints

- Personal, single-user app. No accounts, no server, no Play Store — sideload via
  Android Studio / `adb install`, same as both source apps.
- Android only, package `com.ziv.reminders`, fresh Kotlin/Compose scaffold.
- No data migration from Shape or ReadBook — fresh Room database, all streaks and
  history start at zero.
- Scheduling uses `AlarmManager.setWindow()` (inexact) throughout, plus
  `BOOT_COMPLETED` self-heal and a per-firing "already done?" check — ReadBook's
  proven pattern, not Shape's exact-alarm approach (avoids the Android 14
  `SCHEDULE_EXACT_ALARM` permission fight for a feature that doesn't need to-the-minute
  precision).
- Testing discipline inherits ReadBook's rigor (32 test files) rather than Shape's
  lighter touch (2 test files): TDD for pure logic (including the evaluator's rule
  logic, not just per-kind status/streak math), Robolectric for DAO/AlarmManager
  code, starting at Room schema version 1 and never using
  `fallbackToDestructiveMigration()` on any future version bump (there's nothing to
  migrate *from* on day one — this is future-proofing, distinct from Premise 4's "no
  data migration from Shape/ReadBook"), and a manual on-device QA checklist for
  notification/timer flows Robolectric can't exercise.
- Target minSdk/targetSdk: match ReadBook's (Android 14/API 34 target, given the
  design leans on version-gated behavior — runtime `POST_NOTIFICATIONS` on 13+,
  foreground service type declaration on 14+; exact-alarm permission does **not**
  apply, since this design uses `setWindow()` exclusively, never `setExact()`) —
  confirm exact minSdk floor during planning.
- Notification channels: one channel per `HabitInstance` (not per-kind or shared) —
  the evaluator's escalation (Next Steps #7) works by calling `notify()` again with
  that instance's existing notification ID on its existing channel, never by creating
  a new notification.
- v1 has no in-app "add habit" UI. The three known instances are inserted as seed data
  in the initial Room migration; adding a fourth `HabitInstance` later reuses that same
  seed-data mechanism (a data change in a migration or seeding routine), not a UI —
  consistent with "config, not code," where "config" means a data row, not
  necessarily an end-user-facing screen.
- Shape and ReadBook are manually retired (uninstalled) once Reminders is verified
  reliable on-device — not automatic, and their repos stay on disk for reference. No
  code is deleted from either as part of this project.

## Premises

1. Single-user, local-only, no accounts/server/Play Store — **agreed**.
2. Generic engine, 3 core habit kinds (Counter/Timer/ScheduleCursor) as the
   extensibility primitive — **agreed**.
3. v1 ships exactly the 3 known habit instances (Exercise, Reading timer, Tanakh
   cursor), rebuilt on the new engine, not ported code — **agreed**.
4. No data migration — fresh Room DB, streaks start at zero — **agreed**.
5. Scheduling reuses ReadBook's inexact-alarm + self-heal pattern, not Shape's exact
   alarms — **agreed**.
6. One unified home dashboard replaces three separate single-habit home screens —
   **agreed**.
7. Shape/ReadBook manually retired once Reminders is verified reliable — **agreed**.
8. Testing discipline inherits ReadBook's rigor — **agreed**.

## Cross-Model Perspective

An independent Claude subagent, given a structured summary of this session (no
conversation history), returned this cold read:

- **Coolest unconsidered version:** a rule engine for cross-habit conditions, not just
  a dashboard that displays three habits side by side — e.g. "if Exercise isn't done
  by 12pm and Reading's streak is live, escalate the reading nudge," or "Tanakh
  catch-up count > 5 suppresses the Saturday summary's cheerful tone." The novel move,
  given three sealed kinds already exist, is a small `WorkManager`-based evaluator
  that reads all instance states each morning and composes one notification/action
  set, instead of three independent alarm trains running in isolation.
- **What the session revealed about motivation:** the choice to prioritize
  "add more habit types over time" over widget polish, notification actions, or
  export/backup — an engine-building itch, not a habit-tracking itch.
- **Closest existing project:** Loop Habit Tracker (github.com/iSoron/uhabits, MIT) —
  worth reading its `Habit`/`Streak`/`Reminder` classes for the streak-math and
  reminder-scheduling skeleton, though its single `Habit` entity doesn't have the
  three-way kind polymorphism, the timestamp-delta Doze-immune timer, or the
  ordered-catch-up cursor logic this project needs — none of which exist in any
  known habit tracker.
- **One-weekend build order:** build Counter-with-goal (Exercise) end-to-end first —
  simplest state machine, no foreground service, no CSV asset — to prove the shared
  scaffolding (Room schema, alarm self-heal, dashboard rendering) before the two
  harder kinds build on top of it.

The cross-habit evaluator idea was folded into this design as Approach C (chosen by
the user over the recommended Approach B); the build-order advice (Counter kind
first) is folded into Next Steps regardless of which approach was chosen.

## Approaches Considered

### Approach A: Minimal viable — thin engine, fat instances
Consolidate into one Room schema (`HabitInstance` table with a kind discriminator +
config columns) and one shared AlarmManager/notification module, but keep each kind's
UI screen and completion logic mostly bespoke Kotlin. Fastest to ship, every pattern
already proven 1:1 in ReadBook. Doesn't fully deliver on the "zero-code new instance"
goal — a new instance of an existing kind may still need a few lines of UI wiring.

### Approach B: Ideal architecture — true generic engine
A `HabitKind` sealed interface (Counter/Timer/ScheduleCursor), one `HabitInstance`
Room entity, one generic `HabitScheduler` for all AlarmManager scheduling regardless
of kind, a dispatch layer routing "today's status"/"mark progress" calls to the right
kind's logic, and a dashboard iterating all enabled instances. Actually delivers the
"new instance = one data row, zero code" goal. More upfront design before the first
habit is usable end-to-end; the dispatch layer has no direct precedent in either
source app.

### Approach C: Approach B plus a cross-habit evaluator (chosen)
Everything in Approach B, plus a `WorkManager`-based evaluator — identified by the
independent second opinion — that reads all `HabitInstance` states each morning and
can compose cross-habit conditions (e.g. escalate one habit's nudge based on
another's state). The single most novel capability of the three, at the cost of
stacking a second unproven abstraction (the evaluator) on top of a first (the
dispatch layer) before either has run on-device.

## Recommended Approach

The session's recommendation was **Approach B** — deliver the generic engine first
and prove it end-to-end before adding a second new abstraction layer on top of it, on
the reasoning that Approach C compounds risk rather than retiring it. **The user chose
Approach C instead** — the generic engine plus the cross-habit evaluator, built as one
scope rather than split into a deferred v2. This is a real scope decision made with
the tradeoff explicit, not a default; the build order below sequences the evaluator
last specifically so the three habit kinds and the dashboard are proven and usable on
their own before the evaluator adds cross-habit logic on top.

## Open Questions

- **Evaluator rule set for v1**: the second opinion's examples (escalate a nudge,
  suppress a summary's tone) were illustrative, not decided. The actual v1 rule(s) the
  evaluator should run need to be picked during planning — starting with one concrete
  rule (not a generalized rule-authoring UI) is consistent with the "3 kinds, not a
  fully generic engine" extensibility decision made earlier in this session.
- **Battery-optimization exemption prompt**: both source apps' design docs flagged
  that some OEMs (Samsung, Xiaomi) aggressively kill background alarms without it —
  Reminders will need the same first-launch prompt ReadBook's design called for.
- **Xiaomi/MIUI Autostart**: only relevant if the target phone is MIUI-based; carry
  forward ReadBook's "add a note/deep-link if applicable, skip otherwise" approach.
- **Exact package/app icon/branding polish**: out of scope for this design; cosmetic,
  decide at implementation time.

## Success Criteria

- All three habit instances (Exercise counter, 15-minute reading timer, Tanakh
  chapter cursor) work end-to-end with the exact completion/notification semantics of
  their source apps, rebuilt on the shared engine.
- The unified dashboard shows every enabled habit instance's today-status and streak
  at a glance, matching the approved mockup shape.
- Adding a new instance of an existing kind (e.g. a second timer habit) requires
  inserting one `HabitInstance` row and zero new Kotlin classes.
- The cross-habit evaluator can execute at least one real cross-habit rule connecting
  two of the three shipped habit instances.
- Scheduling survives app kill, phone reboot, and Doze, using the inexact-alarm +
  self-heal pattern — verified on-device, not just in Robolectric.
- Shape and ReadBook are confidently retired from the phone once the above holds for a
  real usage period.

## Distribution Plan

Personal sideload: build a debug/release APK via Android Studio or
`./gradlew assembleDebug` and `adb install` it onto your own phone. No Play Store
listing, no signing pipeline, no CI/CD, no backend — matches both source apps exactly.

## Next Steps

1. Room schema: `HabitInstance` (kind discriminator + per-kind config columns +
   enabled-days mask + notification copy) + per-kind supporting tables reused from the
   source apps' proven schemas (session log for Timer, cursor position for
   ScheduleCursor) + real `Migration` objects from v1 (never destructive fallback).
   "Enabled," wherever it appears later in this doc (dashboard filtering, Success
   Criteria), means the enabled-days mask evaluated against today's date — there is no
   separate instance-level on/off flag in v1, since all three shipped instances are
   permanently present with no add/remove UI.
2. `HabitKind` sealed interface with the three implementations, plus a dispatch layer
   — `HabitEngine`, wrapping one `HabitRepository` per kind (not per instance) —
   routing status/progress calls to the right kind.
   **Spike first**: the three kinds have structurally different completion semantics
   (Counter: single increment; Timer: start/stop timed session via foreground service;
   ScheduleCursor: ordered catch-up cursor). This is the load-bearing abstraction for
   the "zero new Kotlin classes per new instance" success criterion, and the biggest
   technical risk in this design — before Next Steps #3, spend a short spike sketching
   the actual `HabitEngine` interface (e.g. sealed-class status/progress payloads
   varying per kind) so it's proven, not assumed, that one call shape can serve all
   three without leaking kind-specific details into the dashboard/UI layer.
3. Build and verify **Counter-with-goal (Exercise) end-to-end first** — Room through
   UI, `setWindow()` alarm, `BOOT_COMPLETED` receiver, one Robolectric migration test —
   before starting the other two kinds, per the second opinion's build-order advice.
4. Build Timer-with-duration (15-minute reading), reusing the timestamp-delta
   foreground-service model from ReadBook.
5. Build Schedule-cursor (Tanakh), reusing the ordered-catch-up cursor model and CSV
   asset from ReadBook. **Streak semantics** (undefined in either source app, since
   ReadBook's Tanakh feature never had a streak concept): a Schedule-cursor streak
   counts consecutive calendar days with at least one chapter marked read, independent
   of catch-up backlog — falling behind doesn't itself break the streak, only a day
   with zero chapters marked read does. Mirrors Counter/Timer's own "a missed day
   breaks the streak, catching up later doesn't retroactively fix it" rule.
6. Unified dashboard: one screen iterating all enabled `HabitInstance` rows, showing
   today's status + streak per instance, one row per habit, in the shape approved
   during this session:
   ```
   Today
   ✓ Exercise (5/5)
   ⏳ Reading (8/15 min)
   ○ Tanakh (יהושע י"ט)

   Streaks: Exercise 12d | Reading 4d | Tanakh 3d
   ```
7. `WorkManager`-based cross-habit evaluator, built last (after all three kinds and
   the dashboard are independently verified). **Pick the one concrete v1 rule first**
   (see Open Questions), then scope the evaluator's interface to exactly what that
   rule needs — not general rule-authoring infrastructure built ahead of a known need,
   the same YAGNI standard already applied to picking 3 named kinds over a generic
   engine. Before implementing, also settle:
   - **Trigger mechanism/timing**: periodic `WorkManager` job, its interval, and
     target time window (e.g. "once each morning" needs a concrete schedule).
   - **Coordination with `AlarmManager`**: the evaluator and each habit's own nudge
     alarms are independently scheduled systems with different timing guarantees
     (`WorkManager` periodic work has ~15-minute minimum granularity and can be
     deferred by Doze). Rule: the evaluator only updates or escalates an
     *already-fired* notification for an existing habit — it never issues a
     competing/duplicate notification ahead of or racing against that habit's own
     alarm. TDD this rule's logic, same as the other pure-function tests.
8. `POST_NOTIFICATIONS` runtime permission request (Android 13+, first launch, with a
   denied-state banner — same pattern both source apps used); battery-optimization
   exemption prompt (first launch); Xiaomi/MIUI Autostart note if applicable to the
   target device.
9. Manual on-device QA checklist covering all three habit kinds' full-day flows, config
   changes mid-day, notification-permission-denied flow, reboot recovery, and the
   evaluator's cross-habit rule (including the "no duplicate/competing notification"
   coordination rule from #7).

## What I noticed about how you think

- You picked "add more habit types over time" as the 10x version over widget polish,
  notification actions, or export/backup — all of which were real, already-deferred
  ideas from the source apps' own TODOs. You're optimizing for the architecture being
  right, not for feature count.
- When asked how far "config, not code" should go, you didn't take the more abstract
  option — you chose exactly three named kinds over a fully generic rule-authoring
  engine, the same YAGNI instinct visible throughout ReadBook's own design docs
  (rejecting a generalized "any reminder" system in favor of naming Counter, Timer,
  and Schedule-cursor specifically).
- You overrode the recommended approach (B) in favor of C (the cross-habit evaluator,
  pulled into v1 instead of deferred) — a deliberate scope decision made with the
  compounding-risk tradeoff stated plainly, not a default acceptance.
